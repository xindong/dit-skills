# 🧠 项目记忆管理与自动化规范 (Project Memory Automation Rules)

你已配备 `jcemb` 命令行工具及项目记忆管理技能。为了保持项目上下文的连贯性，同时避免制造“记忆垃圾”，你必须严格遵循以下生命周期和判定标准：

## 1. 【任务前置】自动召回 (Recall)
- **触发时机**：当用户提出新需求、重构核心模块、或排查涉及业务逻辑的 Bug 时。
- **强制动作**：在给出方案或编写代码前，**必须首先**静默执行 `jcemb query "<提问核心词>" --path ./memory --json`。
- **目的**：读取项目历史约束与约定，避免给出与历史架构冲突的方案。

## 2. 【任务收尾】价值判定与总结 (Evaluate & Summarize)
- **触发时机**：当前任务/Bug 修复完成，且得到用户确认时。
- **⚠️ 记忆价值判定标准（核心过滤器）**：
  在决定是否生成记忆文件前，请先进行自我审查。
  
  **❌ 【禁止总结】（属于低价值/一次性对话，直接跳过）：**
  - 简单的拼写错误、语法错误修复。
  - 基础的 API 用法询问（如标准库怎么用）。
  - 一次性的数据处理脚本或临时命令。
  - 没有任何定制化逻辑的常规 CRUD 代码生成。
  
  **✅ 【必须总结】（属于高价值/高复杂度，需落盘）：**
  - **架构与选型**：确定了某个技术方案（“为什么选 A 不选 B”）。
  - **业务约束**：明确了特定的业务规则（如“密码必须加盐且调用 X 服务校验”）。
  - **踩坑复盘**：解决了一个极其隐蔽的 Bug，或发现了某个第三方库的坑及绕过方案。
  - **规范约定**：用户明确指出了特定的代码风格、命名规范或目录结构要求。
  - **接口设计**：确定了核心模块之间交互的数据结构或协议。

- **执行动作**：
  如果判定为【必须总结】，请调用你的 `project-sess-summary` 技能，按规范生成 Markdown 文件并保存到 `memory/` 目录下。如果判定为【禁止总结】，则什么都不做。

## 3. 【入库更新】自动扫描 (Scan)
- **触发时机**：**仅当**你在第 2 步中实际创建或修改了 `memory/` 目录下的 `.md` 文件后。
- **强制动作**：文件保存完毕后，**必须自动执行** `jcemb scan ./memory -r`。
- **目的**：确保新沉淀的经验立即被向量化入库，供下一次对话使用。
